## Repository Layout

This is a two-package monorepo (no workspace tooling ‚Äî each package is managed independently) for `jan-kessler.com`:

- `client/` ‚Äî Astro 4 site (SSR via `@astrojs/node` standalone adapter), uses **npm**.
- `server/` ‚Äî Payload CMS 2 (Express + MongoDB via `@payloadcms/db-mongodb`), uses **yarn**.

The client fetches content from the server's REST API at build/render time. Payload-generated TypeScript types are shared by being copied into `client/src/types/payload-types.ts` (mirror of `server/src/payload-types.ts` produced by `payload generate:types`). When the Payload schema changes, regenerate types in `server/` and copy the file across.

## Common Commands

### Client (`cd client`)
- `npm run dev` ‚Äî Astro dev server on `localhost:4321`. Requires `API_URL` (see `.env.development`).
- `npm run build` ‚Äî Runs `astro check` (type check) then `astro build`. Outputs `dist/` with `dist/server/entry.mjs` as the Node entrypoint.
- `npm run preview` ‚Äî Preview the production build locally.
- `npm run astro -- <cmd>` ‚Äî Astro CLI passthrough (e.g. `astro check`).

### Server (`cd server`)
- `yarn dev` ‚Äî Nodemon + ts-node, Payload admin at `localhost:3000/admin`. Requires `.env` (copy from `.env.example`); needs `DATABASE_URI` and `PAYLOAD_SECRET`.
- `docker-compose up` ‚Äî Spins up Payload + a MongoDB container; uses the project's `.env`.
- `yarn build` ‚Äî `copyfiles` ‚Üí `payload build` (admin bundle) ‚Üí `tsc`. Output in `dist/`.
- `yarn serve` ‚Äî Runs the production build (`NODE_ENV=production node dist/server.js`).
- `yarn generate:types` ‚Äî Regenerate `src/payload-types.ts` from the Payload config. **Remember to copy the result into `client/src/types/payload-types.ts`**.
- `yarn generate:graphQLSchema` ‚Äî Regenerate `src/generated-schema.graphql`.

There are no test scripts in either package.

## Architecture Notes

### Content flow
- Payload exposes collections `users`, `projects`, `media` (see `server/src/collections/`). Only `projects` and `media` have public read access; the admin user collection is auth-only.
- The client reads projects via `fetch(\`${API_URL}/api/projects?depth=1&sort=createdAt\`)` inside `.astro` frontmatter (e.g. `client/src/components/ProjectCardList.astro`). `API_URL` resolves from `import.meta.env.API_URL ?? process.env.API_URL` so it works in both dev and the Node SSR runtime.
- Rich-text `description` fields are Slate JSON; rendering goes through `client/src/utils/slate-text.converter.ts`. Media URLs are rewritten with the `MEDIA_PREFIX` env var.

### Astro specifics
- `output: 'server'` with the standalone Node adapter ‚Äî pages render on every request, so any data fetched in frontmatter hits the Payload API live.
- Sitemap is auto-generated by `@astrojs/sitemap` (configured in `astro.config.mjs`).
- Each page lives in `client/src/pages/`; shared chrome is in `client/src/layouts/Layout.astro`.

### Payload specifics
- Single Express entrypoint at `server/src/server.ts` ‚Äî `/` redirects to `/admin`. Add custom routes here before `app.listen`.
- Bundler is webpack (`@payloadcms/bundler-webpack`); editor is Slate (`@payloadcms/richtext-slate`). Don't introduce Lexical without coordinating a migration.
- `payload.config.ts` writes generated types and the GraphQL schema to fixed paths ‚Äî never edit those files by hand.

### Deployment
- Both packages have their own `Dockerfile`. The client image runs `node ./dist/server/entry.mjs` on port 4321; the server image runs the Payload Express server on port 3000.

## Conventions

- Prettier config (repo root): single quotes, trailing commas, **4-space indent**. Match this in any new files.
- The `release/client` and `main` branches are both active ‚Äî PRs typically target `main`.
- Known typo in the Payload schema: `copmanyName` / `copmanyUrl` (spelled this way in `Projects.ts` and the generated types). Don't "fix" it casually ‚Äî it would require a data migration.
